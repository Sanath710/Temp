Section 1

Executive Summary

Greenheck, a global leader in commercial HVAC systems and industrial ventilation solutions, operates a complex manufacturing and distribution network spanning multiple facilities, product lines, and geographic regions. As the organization has grown through organic expansion and strategic acquisitions, the underlying data infrastructure has evolved into a fragmented landscape of legacy data warehouses, departmental data marts, and siloed reporting tools that no longer serve the needs of a modern, data-driven enterprise.

This document presents the complete technical implementation plan for replacing the existing data infrastructure with SAP Datasphere as the unified, cloud-native enterprise data warehouse and analytics platform. The initiative, referred to internally as Project Helix, represents a strategic investment in Greenheck's data capabilities and is a foundational enabler for the company's broader digital transformation roadmap.

Strategic Context

Greenheck's current analytics environment comprises over 15 distinct data repositories, including a legacy SAP BW/4HANA system (version 2.0), multiple SQL Server-based data marts built for individual business functions, and numerous Excel-based reporting processes that persist across operations and finance. This fragmentation results in inconsistent metrics, delayed reporting cycles, excessive manual reconciliation effort, and an inability to perform cross-functional analytics that span the full value chain from raw material procurement through finished goods delivery.

The selection of SAP Datasphere as the target platform was the result of a rigorous evaluation process that assessed six enterprise data platform options against Greenheck's specific requirements for SAP-native integration, semantic layer richness, cloud scalability, total cost of ownership, and alignment with the existing SAP application landscape (SAP S/4HANA, SAP IBP, SAP SuccessFactors).

4.4 Network and Connectivity

Network connectivity is established through SAP BTP Cloud Connector, providing a secure reverse-invoke tunnel without inbound firewall ports. All traffic uses mutual TLS (mTLS), certificate-based authentication, IP whitelisting, and TLS 1.3 encryption. The Data Provisioning Agent operates on a dedicated VLAN with no direct internet access. The DPA cluster consists of 3 active-active nodes (8 vCPU, 32 GB RAM each) behind an HAProxy load balancer within the DMZ.

Section 6

End-to-End Use Cases

6.1 Use Case 1: Order-to-Cash Analytics

6.1.1 Business Context

Order-to-Cash (O2C) is the most critical business process for Greenheck. Currently, analyzing the complete O2C cycle requires extracting data from three separate systems, resulting in a 3–5 day turnaround for any cross-functional analysis.

6.1.2 Business Questions

  • What is the end-to-end cycle time from order receipt to payment collection, segmented by customer, product line, and channel?
  • Where are the bottlenecks in order fulfillment that extend cycle time?
  • What is the order-to-delivery fill rate by product line and plant?
  • What is the revenue leakage from cancelled orders, returns, and pricing discounts?
  • What is the Days Sales Outstanding (DSO) by customer segment?

6.1.3 Key Measures

Table 6.1 — O2C Measure Definitions

6.1.4 Data Flow Configuration

7.3 SAP Integration (OData/CDC)

SAP source systems use the SAP CDC connector, which leverages Operational Delta Queue (ODQ) infrastructure. Key configuration parameters:

  • Subscription Model: Initial load (full snapshot) followed by delta subscription
  • Delta Interval: 60 seconds for near-real-time flows; 15 minutes for batch-adjacent flows
  • Error Handling: Failed delta loads are retried 3 times with exponential backoff (30s, 2min, 10min); persistent failures trigger SAP alert
  • Schema Evolution: Automatic column addition supported; column removal or type change requires manual intervention and re-initialization

7.4 Non-SAP Integration (Data Provisioning)

Non-SAP sources connect through the Data Provisioning Agent (DPA) cluster. Each remote source is configured with connection pooling (min 2, max 8 connections per source) and query timeout (default 300 seconds, configurable per source). JDBC-based sources use prepared statements for parameterized extraction queries. REST API sources use pagination (cursor-based where available, offset-based fallback) with a configurable page size of 1,000–10,000 records.

9.2 Optimization Strategies

  • Partitioning: Analytical datasets are partitioned by date (month-level) to enable partition pruning for time-bounded queries
  • Materialized Views: High-query-frequency aggregations (e.g., daily revenue by product line) are materialized as pre-aggregated tables refreshed every 15 minutes
  • Index Strategy: Primary key indexes on all hash keys and business keys; composite indexes on frequently filtered column combinations identified from SAC query logs
  • Query Federation: Cross-space references use graphical views (federated) rather than physical joins where latency permits, reducing data duplication
  • Delta Processing: CDC-enabled flows use delta loads exclusively after initial load; no full refresh for large tables
  • Connection Pooling: DPA connection pools sized to match concurrent query load; idle timeout set to 300 seconds
